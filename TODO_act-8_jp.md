# TODO act-8: ローカル完結型アーキテクチャへの移行と LLM 実装

このブランチでは、S.S.T システムを「ローカル完結型（エッジ処理）」アーキテクチャへ移行し、`TAGGING_RULE.md` で定義された決定的な LLM メタデータ統合ロジックを実装することに焦点を当てます。

## 1. コアアーキテクチャのリファクタリング
- [ ] **分散パイプラインの廃止**: S3 の `ingest/` プレフィックスを非推奨とし、主要な処理パスから Prefect フローへの依存を削除します。
- [ ] **Worker 機能をローカルサービスへ統合**: `AudioTagger`、`MusicBrainzIdentifier`、およびメタデータ抽出ロジックを Scout（ローカルプロセッサ）に直接統合します。
- [ ] **ローカル状態管理**:
    - [ ] 処理済みの `app_id`、キャッシュされたメタデータ、およびローカルファイルのハッシュを追跡するための SQLite データベースを実装します。
    - [ ] `--force` フラグが使用されない限り、処理済みのアルバムをスキップするようにします。

## 2. 音声処理と採用ロジック
- [ ] **マルチフォーマット・メタデータ抽出**:
    - [ ] 変換を行う前に、*すべての* 音声ファイルバリアントからタグを抽出するスキャナーを実装します。
    - [ ] すべてのファイル（特に WAV）について、正確な再生時間（秒単位）を計算します。
- [ ] **厳格な採用優先順位**:
    - [ ] ロスレス (FLAC/WAV/ALAC) -> AIFF (24bit/48kHz) へ変換。
    - [ ] 高品質ロッシー (AAC/OGG) -> MP3 (CBR 320k/48kHz) へ変換。
    - [ ] MP3 -> そのまま採用（再エンコードなし）。
- [ ] **トラックごとのアートワーク**: トラック単位でアートワークを抽出・埋め込みを行うようにロジックを更新します。

## 3. LLM メタデータ統合（「整理者」）
- [ ] **プロンプトエンジニアリング**:
    - [ ] 「事実ベースのメタデータ整理者」としての LLM 用システムプロンプトを作成します。
    - [ ] 厳格な指示を含めます：ハルシネーション（推論）の禁止、言語優先度（設定 > 英語 > 原語）の遵守。
- [ ] **コンテキストの構築**:
    - [ ] フォーマット別タグセット、Steam API データ、および MusicBrainz データを LLM に渡します。
    - [ ] WAV ファイルについては、再生時間比較に基づく MusicBrainz のマッチング結果を渡します。
- [ ] **競合の解決**: 複数のソース間で最も詳細かつ一貫性のある文字列を LLM が選択するためのロジックを実装します。

## 4. Review とアーカイブのワークフロー
- [ ] **最終パッケージング**:
    - [ ] `metadata.json` と `.acf` を含む `archive/` へのアップロードを標準化します。
    - [ ] メタデータとともに LLM とのやり取りログ（プロンプト/回答）を保存します。
- [ ] **強化された Review バンドル**:
    - [ ] 信頼度が低い場合、採用音源、`.acf`、生メタデータソース JSON、および LLM ログを含むバンドル ZIP を作成します。
    - [ ] バンドルを `review/` にアップロードします。

## 5. UI と統合
- [ ] **バックエンドの更新**: ダッシュボードのレンダリングを高速化するため、新しい SQLite キャッシュから読み取るように UI バックエンドを調整します。
- [ ] **ログビューア**: S3 に保存された LLM 統合ログを UI で表示できるようにします。
