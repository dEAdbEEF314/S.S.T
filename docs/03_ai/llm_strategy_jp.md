# LLM 戦略: 事実に基づくメタデータ整理

SSTは、断片的なソースから収集されたメタデータの統合と正規化のためにLLMを使用します。

## 1. 基本原則

- **ゼロ・ハルシネーション**: 推論は禁止されています。不足しているデータには "Unknown" を使用します。
- **逐次統合**: スケーラビリティのためにトラック単位で処理します。
- **サマリー・ファースト・パス (実装予定)**: 逐次ループの前に、アルバム全体のトラックリスト概要をLLMに送信し、アーティスト名やディスク枚数の一貫性を確立します。
- **コンフリクト・レポート**: 高度なメタデータの競合が検出された場合、LLMは競合情報（URL、一致しないフィールド等）をまとめ、ユーザーのために `review/` キューへ転送します。

## 2. 実装: マルチモデル互換性

SSTは特定のモデルに依存しません。コンテキストウィンドウの広さから Gemini 1.5 Pro を推奨していますが、**OpenAI Chat Completions API** をサポートするプロバイダーであれば、以下を含めすべて使用可能です：
- Google Gemini (OpenAI互換レイヤー経由)
- OpenAI GPT-4o
- DeepSeek / Groq / ローカルLLM (OpenAI互換エンドポイントを持つ場合)

## 3. 競合解決とレビュー

LLMが高精度で解決できない競合に遭遇した場合：
1.  `conflict_report.json` を生成します。
2.  レポートには具体的な詳細（「MBZタイトル: A vs Steamタイトル: B」、「アーティストURLの不一致」など）が含まれます。
3.  アルバムはこのレポートと共に `review/` に移動され、ユーザーの手動修正を支援します。

## 4. 優先順位と重み付け

ユーザーは設定ファイルでメタデータソースの優先順位を構成できます：
`METADATA_SOURCE_PRIORITY=MBZ,STEAM,EMBEDDED` (デフォルト)

システムは出所に基づいて一致を区別します：
- **確定扱い (Confirmed)**: 特定の「Digital Media」形式、および「Bandcamp」以外のソースに基づく一致。
- **弱い情報 (Weak)**: トラック数や日付の近さのみに基づく一致。
