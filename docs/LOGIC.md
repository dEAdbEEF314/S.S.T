# S.S.T (Steam Soundtrack Tagger) メタデータ判定ロジックとアーキテクチャ

本ドキュメントは、S.S.Tがどのようにしてメタデータを収集・評価し、最終的なタグ付けとアーカイブ判定（Archive / Review）を行っているかのコアロジックを定義します。

## 1. 三権分立のアーキテクチャ

S.S.Tは、AIの推論の柔軟性と、プログラムの厳格な正確性を両立させるため、**「三権分立」**のアーキテクチャを採用しています。

| 役割 | 担当 | 責務と行動内容 |
| :--- | :--- | :--- |
| **立法** | **User (ユーザー)** | `.env` の `METADATA_SOURCE_PRIORITY` を通じて、「どのソースを最も信頼するか」という絶対的な基準（憲法）を定義します。 |
| **司法** | **LLM (マスター・アーカイブ監査官)** | 提示された複数のソース（証拠）を比較し、ユーザーが定めた優先順位に基づき、最も妥当な曲名やアーティスト名を推論します。システムに誘導されるのではなく、音楽的知識と文脈から独立した「判決」を下します。 |
| **行政** | **System (Pythonロジック)** | LLMが出力したメタデータ（判決）が、DJ機材での運用に耐えうる物理的基準（クリーンネス）を満たしているかを最終検閲します。合格すれば `Archive` を執行し、基準を満たさなければ `Review` へ強制送行します。 |

---

## 2. METADATA_SOURCE_PRIORITY と「ユーザが求める正しさ」

ユーザーは `.env` ファイルにて、以下のソースの優先順位を定義できます。
*(例: `METADATA_SOURCE_PRIORITY=STEAM_PICS,STEAM_STORE,MBZ,STEAM_TAGS,EMBEDDED`)*

*   **STEAM_PICS**: Steam内部の秘匿データ（通常、最も正確で純度が高い）
*   **STEAM_STORE**: Steamストアページに表示される公式トラックリスト
*   **MBZ**: MusicBrainz (コミュニティベースの音楽データベース)
*   **STEAM_TAGS / EMBEDDED**: ダウンロードされたファイルに最初から付与されているタグやファイル名

### LLMへの重みづけと指示
システムは、この `METADATA_SOURCE_PRIORITY` をそのままLLMのプロンプトに注入します。
LLMには**「上位のソースを絶対的に優先せよ。ただし、上位ソースに『01 - 』のようなトラック番号が混入している場合は、下位のクリーンなソースと比較し、純粋な曲名のみを抽出（クリーニング）せよ」**という高度な重み付けと自浄作用の判断が求められます。

## 3. MusicBrainz 候補のスコアリングロジック (NWO Hybrid Scoring System)

MBZから取得した候補は、以下の独自のスコアリングシステムによってランク付けされ、上位5件がLLMに提示されます。

| 評価階層 (Tier) | 判定基準 | 加点 / 減点 |
| :--- | :--- | :--- |
| **Tier 1: 決定論的証拠** | Steam公式ストアリンク (`store.steampowered.com/app/{AppID}`) の存在 | **+500** |
| | SteamDBリンク (`steamdb.info/app/{AppID}`) の存在 | **+500** |
| | 親ゲームのSteam/SteamDBリンクの存在 | **+300** |
| | Bandcampリンクの存在 | **+100** |
| **Tier 2: 意味論的・構造的証拠**| アルバムタイトルの文字列類似度 (Steam Store または Localタグと比較) | **最大 +100** |
| | トラック数の完全一致 | **+50** |
| | トラック数の不一致 (差分に基づくペナルティ) | **最大 -300** |
| **Tier 3: 状況証拠** | リリースフォーマットが `Digital Media` である | **+30** |
| | リリース年の完全一致 | **+20** |
| | リリース年の不一致 (差分に基づくペナルティ) | **最大 -100** |
| **Tier 4: トラックリストの指紋** | ローカルファイルの曲名とMBZの曲名が **80%以上** の割合で類似一致 | **+200** |

*注意点*: 正解のMBZデータであっても、MBZの編集者がSteamのURLを関連付けていない（Tier 1のボーナスが得られない）、あるいはトラックリストの表記揺れによりFingerprintマッチ（Tier 4）を逃すと、スコアが100前後に留まり、順位が低下するケースが発生します。
*(追記)*: 過去のバージョンにおいて、同一AppID内にAIFFとMP3等の複数フォーマットが混在する場合、ローカルトラックのタイトルとして「1」「2」といったトラック番号のみが誤って抽出されてしまい、Fingerprintマッチ（MBZの曲名との照合）が失敗してスコアが100に留まる（AppID: 1040700等のケース）という問題がありました。現在はローカルベースライン抽出処理において、タグやファイル名から実際の文字列（曲名）を正しく抽出し、正確なFingerprint照合が行われるように修正されています。

## 4. LLMに求めている「判断」

LLMは単なるパーサーではなく、以下のスコア付けとデータの再構築を行う「監査官」として機能します。

1.  **Identity Confidence (0-100)**
    *   Steamの公式情報と、MusicBrainzの候補、ローカルファイル群が **「完全に同一の作品を指しているか」** の確信度。
    *   別アーティストの同名アルバムや、情報が部分的にしか一致しない場合は、LLMが自律的にスコアを下げ、安全のためにReviewを促します。
2.  **Integrity Quality (0-100)**
    *   メタデータの品質。欠損がないか、ノイズ（不要な文字）が含まれていないかの評価。
3.  **タイトルクリーニング (Absolute Mandate)**
    *   いかなるソース（たとえMBZの公式名）であっても、タイトルの先頭にある数字（例: `01. `）は「DJ機材での視認性を落とすノイズ（Dirty Tags）」とみなし、純粋な曲名（`override_title`）を生成する。

## 4. システムの挙動と最終検閲（セーフティネット）

LLMが推論を終えた後、システム（`processor.py` 及び `builder.py`）が介入し、厳格なルールベースの検証と補正を行います。

### 4.1 強制クリーニングロジック
LLMが番号の除去を漏らした場合や、ソースの汚染が強固な場合に備え、メタデータを書き込む直前にシステムが正規表現を用いて強制クリーニングを行います。
*   **検知パターン**: `^(\d+)([\s.-]+)` (行頭の数字とそれに続く区切り文字)
*   **処理**: マッチした部分を強制的に削ぎ落とし、クリーンなタイトルを生成します。

### 4.2 Review 判定の厳格なゲート
以下のいずれかの条件に該当した場合、LLMが「100点」を出していても、システムは強制的に `Review` 判定を下します。

1.  **Dirty/Conflicting Tags の検知**: 強制クリーニングが発動した形跡がある場合、またはLLMが不純な文字を残した場合。
2.  **物理的欠損 (Track #0)**: ファイル名やソースからトラック番号が特定できず、`0` にフォールバックした場合。
3.  **品質スコアの不足**: Identity Confidence が `100` 未満、または Integrity Quality が `95` 未満の場合。
4.  **音質警告**: 音源ファイル自体に問題（例：低ビットレートなど）がある場合。

### 4.3 Fast-Track (決定論的処理)
コスト削減と高速化のため、システムが「Steamストアの情報とMusicBrainzの情報が完全に一致し、曲数やタイトルにブレがない」と判断した場合、LLMによる推論プロセスを完全にバイパスし、システム主導で安全にアーカイブ処理を完了させます。

## 5. パッケージングと成果物の出力仕様

S.S.Tは、処理された音声ファイルとメタデータ、および詳細な処理ログを一つのパッケージとして出力します。

### 5.1 中間生成物としてのZIP
WSL2（Linux環境）とWindowsマウント先（ホスト環境）の間でのファイルコピーの信頼性を担保するため、システムはまずLinux側の高速な一時ディレクトリで成果物を生成し、それを **ZIPアーカイブ** として一つのファイルにまとめます。

### 5.2 Windowsへの転送と展開
ZIP化されたパッケージは、設定された `SST_OUTPUT_DIR`（通常はWindows側のパス）へ移動されます。転送完了後、システムはWindows標準の `tar.exe` を呼び出してその場で展開（Unzip）を行います。

### 5.3 最終的な出力状態
*   **最終成果物**: ZIPと同名の **ディレクトリ** が作成され、内部に展開されたファイルが配置されます。
*   **クリーンアップ**: 展開が正常に完了したことを確認した後、**中間ファイルであるZIPはシステムによって自動的に削除されます。** ユーザーの手元に残るのは、すぐにライブラリへ移動可能な展開済みのディレクトリのみです。

---

## 6. まとめ

S.S.Tの判定ロジックは、**「AIの知性（LLM）によって荒れたソースから正解を推測し、プログラムの規律（システム）によって徹底的にノイズを排除する」** という双方向の防衛線によって成り立っています。このアーキテクチャにより、「一度のアーカイブで、一生の信頼を」という S.S.T の理念がデータレベルで保証されます。
