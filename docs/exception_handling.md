# 例外処理 (Exception Handling)

本ドキュメントでは、通常のメタデータ・マッチングルールやバリデーション処理から大きく逸脱する特殊なケース（「特級呪物」と呼称）に対するシステム側のフォールバック処理と、対応指針について定義します。

## 1. 同一ディレクトリへの複数OSTインストールとメタデータ欠落

### 1.1. 現象・課題
* **事象**: 複数の独立したAppID（DLC等）の音源データが、開発者の設定によりSteam上で完全に同一のインストールディレクトリ（例: `steamapps/music/Happy's Humble Burger Farm OST`）に配置されてしまうケースがあります。
* **メタデータ欠落**: この際、対象のMP3などの物理ファイルに「アルバム名」等のタグ情報が著しく欠落している（空欄になっている）場合、S.S.TのシステムはこれらのファイルをAppIDごとに事前に分割・隔離することができません。
* **結果**: 1つのAppIDを処理する際に、無関係な他のAppIDのファイルも含めた**全てのファイル（百曲以上など）**が処理対象として一括で読み込まれてしまいます。一方で、STEAM等の公式参照データには、そのAppID本来の曲数（例: 10曲）しか登録されていないため、以下のような問題が発生します。
    * 「異常な数の重複（Duplicates）」が検出される。
    * LLMのCoherenceマッピングにおいて、後半のチャンクを処理する際に参照データが枯渇し、対応インデックスとして `null` を返却する。

### 1.2. フォールバック処理とシステム挙動
このような極端に破綻したメタデータを持つタイトルに対して、システム側で無理に正確なマッピングを推論（オーバーエンジニアリング）することは試みません。基本的には以下のフォールバック処理に従い、システムをクラッシュさせることなく安全に `Review` へ送ります。

1. **LLMの参照枯渇時の安全なハンドリング (`src/sst/llm.py`)**:
   LLMが Coherence Map-Reduce フェーズにおいて、参照アルバム側のトラックが枯渇したことを正しく検知し、インデックスとして明示的な `null` を返してきた場合、システムはこれを型エラー（`TypeError`）としてクラッシュさせるのではなく、「対応する参照トラックが存在しない（空リスト `[]`）」として安全にフォールバックさせ、マッピング処理を継続します。
2. **Review判定の許容**:
   このフォールバックの結果として、対象のアルバムには大量のトラックが「ローカル専用トラック」または「重複（Duplicates）」としてアサインされたまま残存します。
   この状態は、その後のバリデーション処理（`ResultValidator` による重複曲の存在チェックや、Qualityスコアの低下など）に確実に抵触するため、最終的なステータスは自動的に `Review` と判定されます。
3. **運用上の正解**:
   メタデータと物理ファイルの配置ルールがともに著しく崩壊しているタイトルについては、この `Review` 判定に流し込むことこそがシステムとしての「正しい終着点（正解）」であり、自動解決の対象外（手動介入またはスキップ対象）とします。
