# Fingerprint Test Analysis Report (2026/06/02)

## 概要
指紋照合（Fingerprint）を有効にした環境での100件規模テスト（サブセット）の結果を分析し、LLMの判定意図とシステムロジック（最終判定）の間に生じた乖離を分類した。

## 乖離パターンの分類

### 1. 【システム救済型】物理整合性によるReview強制
LLMがArchiveを推奨したものの、システム側でReviewへ落としたケース。

*   **事例:** `Happy’s Humble Burger Farm: Score (OST)`
*   **挙動:**
    *   **LLM判定:** `archive_vs_review_ratio: 100:0` (Archive推し)
    *   **最終判定:** **Review** (`[Duplicates (83), Quality too low (85%)]`)
*   **分析:** 
    LLMは「指紋照合率（80%超）」を重視してArchiveと判定したが、マッピングの結果、83箇所のトラック番号重複（Collision）が発生した。システムバリデーターが「1ファイル1トラック」の物理原則に反すると判断し、LLMの意図をオーバーライドした。
*   **教訓:** 指紋が一致していても、マッピングロジック（特に大量のバリエーション曲を持つアルバム）での衝突回避が必要。

### 2. 【フォーマット欠落救済型】暗黙的信頼のArchive完遂
LLMが必須の比率パラメータを返さなかったが、システムが文脈を読み取ってArchiveとしたケース。

*   **事例:** `GRAND CROSS: ReNOVATION Original Soundtrack`
*   **挙動:**
    *   **LLM判定:** `archive_vs_review_ratio: 0:0` (未回答)
    *   **最終判定:** **Archive** (`Success [Steam Trust]`)
*   **分析:**
    LLMは判定比率を空で返したが、`confidence_score: 100` および `confidence_reason` 内で「SteamとLocalの構造的一致により信頼する」と明記していた。システム側は数値を優先し、マッピングに矛盾がなかったためArchiveとして処理を完遂した。
*   **教訓:** LLMの出力フォーマットに揺れがあっても、自信スコアと理由が明確であればシステムが補完可能。

### 3. 【構造的乖離型】インデックス・スライドによる不一致
LLMは「一致」と確信したが、マッピングのズレによってReviewとなったケース。

*   **事例:** `Dead Estate Soundtrack`
*   **挙動:**
    *   **LLM判定:** `Confidence: 100`, `Quality: 95` (確信)
    *   **最終判定:** **Review** (`[Duplicates (15)]`)
*   **分析:**
    Steam側のリスト番号とLocalのファイル名（番号）が1つずれて開始されていた。LLMは「順序が同じなので100%一致」と判断したが、システムによる逐次マッピングの結果、末尾で不整合が発生。
*   **教訓:** トラック番号の開始位置（0開始 vs 1開始）やプレフィックスの解釈における、LLMとロジックの「認識のズレ」がReviewの原因となる。

## 今後の改善提言

1.  **Smart Rescue Logicの活用:** `Happy's Humble` のような大量重複ケースでは、今回強化した「重複解消ロジック」を適用し、インデックスのスライドを自動補正できるか検証が必要。
2.  **LLMプロンプトの微調整:** 判定比率（ratio）の出力を徹底させるための、Few-shot例の強化。
3.  **スライド検知:** トラック番号が一定数連続して「1つ隣」とマッチしている場合、オフセットを自動計算するロジックの検討。
