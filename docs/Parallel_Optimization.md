# アルバムサイズに応じた並行処理の最適化案

## 1. 目的
安定性の高い `qwen2.5:7b-sst` モデル（32Kコンテキスト）を全ティアで固定使用することを前提に、アルバムサイズ（ティア）に応じて並行処理数（ワーカー数）を動的に変更することで、精度を維持したまま全体の処理時間を短縮します。

## 2. 調査結果

### 2.1 現状のボトルネック
*   **LLM 推論**: 7B モデルは 16GB VRAM において比較的軽量ですが、巨大なアルバムのコンテキスト処理には時間がかかります。
*   **AcoustID スキャン**: 全曲指紋照合を有効にした場合、1曲ごとに 1.0〜1.5秒の API ウェイトが発生するため、100曲のアルバムではスキャンだけで数分を要します。
*   **直列処理**: 現在の MEDIUM / LARGE ティアはワーカー数 1 で固定されており、前のアルバムが終わるまで次のアルバムの指紋照合すら開始されません。

### 2.2 最適化の余地
*   **並列スキャン**: AcoustID のスキャンは I/O 待ちがメインであるため、複数のアルバムを並行してスキャンすることで、ウェイト時間を有効活用できます。
*   **VRAM の有効活用**: 7B モデルであれば、2〜3 並列程度の LLM 推論は 16GB VRAM 内で（KVキャッシュを考慮しても）実行可能です。

## 3. 提案：ティア別動的ワーカー割り当て

アルバムの曲数に応じて、以下のように並行処理数を調整します。

| ティア | 対象曲数 | 推奨ワーカー数 | 理由 |
| :--- | :--- | :--- | :--- |
| **SMALL** | <= 50 | **4 〜 6** | 個々のコンテキストが小さく、並列スキャンによる高速化効果が最大。 |
| **MEDIUM** | 51 - 100 | **2** | コンテキスト増大に伴う VRAM 負荷を考慮し、バランスを重視。 |
| **LARGE** | > 100 | **1** | 最大の安定性と VRAM 空間を確保するためシリアル処理を維持。 |

## 4. 実装上の課題と解決策

### 4.1 AcoustID API のレート制限 (1 req/sec)
複数のアルバムが同時に指紋照合を開始すると、API 制限に抵触するリスクがあります。
*   **解決策**: `AcoustIDIdentifier` クラスに、プロセス全体で共有されるセマフォまたはトークンバケット型のレートリミッターを導入し、全スレッド合計で「1秒に1リクエスト」を超えないように制御します。

### 4.2 Ollama の並行リクエスト設定
Ollama はデフォルトで並行リクエストを制限している場合があります。
*   **解決策**: サーバー起動時の環境変数 `OLLAMA_NUM_PARALLEL` を 4 以上に設定することを推奨事項としてマニュアルに追加、またはセットアップスクリプトで設定します。

### 4.3 VRAM 溢れ（OOM）の防止
大規模なアルバムが複数同時に LLM 推論に入ると VRAM が不足する可能性があります。
*   **解決策**: `llm.py` の `DistributedRateLimiter` を拡張し、同時実行中のトークン総量に基づいて推論の開始を待機させる「スロットリング」機能を検討します。

## 5. 期待される効果
*   小型アルバムが多数ある環境において、総処理時間を **50% 〜 70% 短縮** できる可能性があります。
*   大型アルバムの処理中であっても、小型アルバムの「先行スキャン」を許容することで、パイプラインの停滞を最小限に抑えます。
