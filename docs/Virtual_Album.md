# 仮想アルバム (Virtual Album) 構想：物理的確証に基づくメタデータ監査

## 1. 概要と哲学

S.S.T (Steam Soundtrack Tagger) の仮想アルバム構想は、以下の根源的な問いからスタートしています。
**「音声指紋が一致したのなら、その曲の『物理的な正体（タイトルとアーティスト）』は、世界のデータベースですでに確定しているはずではないか？」**

本構想の目的は、LLMに不確実な推論を強いるのではなく、**「物理的な事実（FINGERPRINT）」を主軸に据え、そこに「販売上の名称（STEAM）」や「手元の状態（LOCAL）」をマッピングする**という、極めて決定論的かつ知的な監査プロセスを構築することにあります。

LLMは「何が正しいか」をゼロから考えるのではなく、Python側で整理された「確定事実」の最終的な登記・調整役（Auditor & Adjuster）として機能します。

## 2. 4つの仮想アルバムと情報の重み付け

1.  **FINGERPRINT 仮想アルバム (Ground Truth / 物理的真実)**:
    *   **役割**: システムの核。音声波形に基づき、MusicBrainz (MBZ) から取得した「物理的な実体」としての情報。
    *   **確定情報**: トラック名、アーティスト、詳細クレジット（作詞・作曲・編曲等）、ISRC、ディスク構造。
    *   **信頼性**: 物理的波形が一致している場合、この情報を「絶対的な正解候補」として扱います。
2.  **MBZ_SEARCH 仮想アルバム (Semantic Truth / 意味的真実) [NEW]**:
    *   **役割**: 音響指紋（波形）が未登録の場合の強力な代替手段。SteamのAppID（URL）やアルバム名を用いた直接検索・テキスト検索によるハイブリッドスコアリング最上位候補。
    *   **特徴**: SteamストアURLとの紐付け（直接逆引き）を最優先とし、一定スコア未満のノイズは足切り（破棄）されます。また、FINGERPRINTと同じMBIDを指し示した場合は「v_verified_mbz」として統合され、100%の確証として扱われます。将来的にVGMdb等への拡張枠（v_web_search）としても機能します。
3.  **STEAM 仮想アルバム (Contextual Info / 販売上の文脈)**:
    *   **役割**: Steamストアにおける「商品」としての情報。
    *   **用途**: 公式の作品名、販売元情報、およびバイリンガル（特に日本語）タイトルの補完。MBZ_SEARCHと構造が一致した場合、強い信頼性を獲得します。
4.  **LOCAL 仮想アルバム (User Hints / 手元の証拠)**:
    *   **役割**: ユーザーの手元にあるファイル名やフォルダ名。
    *   **用途**: FINGERPRINT が複数候補（通常盤/インスト盤）で同票になった際の「判別ヒント」として利用。
    *   **曲順決定とソート仕様**:
        *   ローカルファイルのディスク番号および曲順決定には、音源ファイル内の埋め込みメタデータ（タグ情報）を最優先で参照する。
            *   **ディスク番号 (`disc`)**: 埋め込みタグの `disc_number` を最優先で参照する。タグが存在しない場合は、フォールバックとして格納フォルダ名（`disc_N` や `CD_N`）から数値を抽出する。
            *   **トラック番号 (`track_num`)**: 埋め込みタグの `track_number` を最優先で参照する。タグが存在しない、または不備がある場合は、フォールバックとしてファイル名の先頭から抽出したトラック番号数値（`filename_track`）を利用する。
        *   これらにより決定されたディスク番号（`disc`）とトラック番号（`track_num`）を基準に、LOCAL仮想アルバム内のトラックリストを昇順ソートしてLLMへ提示する。これにより、ディスクをまたいだ順序の不一致や、平置きされた複数枚組アルバムでの誤爆（同一曲の重複認定）を防止する。

## 3. FINGERPRINT: 物理的確証の最大化ロジック

### 多数決とスコアリング
アルバム全曲の指紋を用いて以下の「多数決（クロスバリデーション）」を行い、Release ID を一本釣りします。

1.  **物理一致スコア**: AcoustID 照合率と Duration（再生時間）の近似（±3秒以内）で足切り。
2.  **構成一致加点**: トラック数や並びがローカルと一致する場合に高加点。
3.  **キーワード加点**: ローカルのファイル名に含まれる `Instrumental` 等の単語と、MBZ側のリリース情報を照合。

### 詳細クレジットの取得
確定した Release ID に基づき、MusicBrainz API の `relationships` を深掘りし、以下の詳細データを各トラックに付与します。
*   **制作陣**: Composer（作曲）, Lyricist（作詞）, Arranger（編曲）, Remixer（リミキサー）。

## 4. カバーアートの優先順位（実利重視）

DJタグとしての利便性を考慮し、以下の階層構造で決定します。
1.  **第1優先 (Embedded)**: 元のローカルファイル（Steam配布音源）に埋め込まれている画像。トラックごとに異なる公式アートワークがあるケースを尊重します。
2.  **フォールバック (Cover Art Archive)**: FINGERPRINT (MBZ) に紐づく公式ジャケット画像。
3.  **最終手段 (Steam Store)**: Steamストアのバナー画像。

## 5. LLMの役割：判別と調整への特化

LLMはもはや「どのMusicBrainz候補が正解か」を迷う必要はありません。そのタスクは以下の3点に純化されます。

1.  **候補の判別**: FINGERPRINT の多数決が同票（通常盤 vs インスト盤など）の場合、LOCAL のヒントを見て最終決定を下す。
2.  **表記の融合**: FINGERPRINT の正確なクレジットと、STEAM の作品名としての名称を、バイリンガル設定等に従って最適にブレンドする。
3.  **最終調整**: タイトル末尾の余計な記号やトラック番号のクリーニングを行い、DJ機材での視認性を最大化する。

## 5. 実装上の工夫と解決された課題

プロトタイプ実装の過程で、大規模なアルバムにおけるLLMの安定性に関する以下の課題を特定し、解決しました。

*   **コンテキストサイズとVRAMの最適化**:
    Ollamaのコンテキストウィンドウを32,768（32K）に拡大し、プロンプト内のJSONデータから冗長なフィールドを削除することで、100曲近いアルバムでも情報の欠落（Truncation）を防ぎました。
*   **Direct JSON出力の強制**:
    最近のLLM（Qwen 3.5等）が過剰に「思考（Thinking）」を行い、出力バジェットを使い果たす問題を解決するため、システムプロンプトを「メタデータ監査エンジン」として純粋なJSON出力に特化させ、推論ブロックを抑制しました。
*   **安定モデルの選定**:
    当時のプロトタイプ検証では `qwen2.5:7b` を主要な評価対象として使用しました。現行実装の既定モデルは固定ではなく、`.env` / `src/sst/config.py` の `LLM_MODEL` と `LLM_BACKEND` によって切り替え可能です。

## 6. 現在の実装との比較・期待される効果

| 項目 | 現在の実装 (`src/sst/`) | 仮想アルバム構想導入後 |
| :--- | :--- | :--- |
| **同定 (Identity) の責任** | LLM（複数のMBZ候補から推論・選択） | Pythonロジック（AcoustIDクロスバリデーションで確定） |
| **LLMへの入力情報** | 大量のMBZ候補リストとSteam生データ | 完全に構造化され、足切りと統合が行われた最大4つの仮想アルバム情報のみ |
| **LLMのタスク** | 同定とタグマージの同時処理 | フィールドごとのタグマージと最終調整のみ |
| **コンテキストサイズ** | 巨大（100曲超でトークン溢れの危険） | 軽量（確定情報のみを渡すため） |
| **処理精度・安定性** | 複数候補による迷いやハルシネーションが発生しやすい | 決定論的アプローチにより誤爆が理論上ほぼゼロになる |

### 結論

仮想アルバム構想の導入により、LLMの推論負荷とプロンプトサイズを劇的に削減でき、同時に「FINGERPRINTによる多数決」という決定論的ロジックによってシステムの同定精度と安定性が飛躍的に向上します。LLMは本来の強みである「表記揺れの吸収」や「優先順位に基づく最適なフィールド選択」に専念できるようになります。
