# 仮想アルバム (Virtual Album) 構想

## 1. 概要と目的

S.S.T (Steam Soundtrack Tagger) における現在のメタデータ同定・統合プロセスは、LLMに不確実な候補リスト（MusicBrainzの複数候補など）と生データを直接渡し、「同定（正解の決定）」と「メタデータのマージ」を同時に行わせています。
これが、巨大なアルバムにおけるプロンプトのコンテキスト肥大化や、バリエーション曲に対する推論ミス（ハルシネーション等）の根本原因となっていました。

**仮想アルバム構想**は、この「同定」の責任をPython（ロジック）側に移し、LLMのタスクを「フィールド単位での最適な採用とマージ」に特化・単純化するための新しいアーキテクチャです。

本構想では、情報源ごとに完全に構造化された「仮想アルバム（Virtual Album）」を3種類作成し、LLMはそれらを比較して最終的なタグ情報を決定します。

## 2. 3つの仮想アルバム定義

`docs/TAGGING_RULE.md` で定義されたフォーマットに従い、以下の3つの仮想アルバムをPythonロジックで事前構築します。取得できない情報は `null` とします。

1. **STEAM 仮想アルバム**:
    *   **情報源**: Steam Web API (基本情報) および Steam PICS (詳細トラックリスト・クレジット等)
    *   **特徴**: Steam上での正式名称や、パブリッシャー・デベロッパー情報、公式の言語タグを含みます。
2. **FINGERPRINT 仮想アルバム**:
    *   **情報源**: AcoustID (指紋照合) + MusicBrainz (MBZ)
    *   **特徴**: 音声波形に基づく物理的な確証。アルバム全体の指紋を用いた「多数決（クロスバリデーション）」により、極めて高い精度で同定されたメタデータを含みます。
3. **LOCAL 仮想アルバム**:
    *   **情報源**: ローカルファイルのメタデータ (ID3等)、ファイル名、ディレクトリ名など
    *   **特徴**: ユーザーが手元に持っている元々の情報。他の情報源が不足している場合のフォールバックや、マイナーなバリエーションの判定に利用します。

## 3. FINGERPRINT: 最高の確からしさを実現するパイプライン

FINGERPRINT仮想アルバムの構築においては、1曲単位のスコアに依存するのではなく、**アルバム全曲の指紋を用いた多数決（クロスバリデーション）**を行うことで、同定精度を飛躍的に高めます。

### ステップ定義

1.  **全曲一斉スキャン**: アルバム内の全楽曲の音声指紋（Chromaprint）を AcoustID API に送信し、関連する `Release ID` 候補を全曲分取得する。
2.  **指紋群による多数決 (クロスバリデーション)**:
    *   曲ごとに返ってきた `Release ID` の出現回数をカウントする。
    *   全曲を通じて「最も多く登場した共通の Release ID」を正解のターゲットアルバムとして一本釣り（確定）する。
    *   *メリット*: 1曲では判別できないシングル盤とアルバム盤の違いも、アルバム全体の構成（トラック数や長さの並び）で確実に見分けることができます。
3.  **MusicBrainz API の直接フェッチ**:
    *   確定した `Release ID` をベースに本家 MusicBrainz API (または `musicbrainzngs`) を叩き、`recordings`, `artists`, `labels`, `tags`, 各種クレジット (`relationships`) のフルデータを取得する。
4.  **Cover Art Archive からのジャケット取得**:
    *   確定した `Release ID` に紐づく Cover Art Archive API を叩き、高画質なフロントカバー画像を確定で取得する。

## 4. 精度向上案（Python側での追加ロジック）

*   **Tie-break（同票時の決着）ロジック**:
    FINGERPRINTの多数決で同票（例：通常盤とインスト盤）になった場合、LOCALのファイル名（例: "Off Vocal", "Instrumental"）の有無や、ローカルの総トラック数との一致率をPython側で評価し、自動で正解を決定（Tie-break）する。
*   **Duration（再生時間）マッチング**:
    MusicBrainzのデータとローカルファイルをマッピングする際、単純なトラック番号だけでなく、「再生時間（Duration）の近似（±2秒以内）」を紐付け条件に加える。これにより、メタデータが一切存在しない匿名のローカルファイルでも完璧なバインドが可能になる。

## 5. 実装上の工夫と解決された課題

プロトタイプ実装の過程で、大規模なアルバムにおけるLLMの安定性に関する以下の課題を特定し、解決しました。

*   **コンテキストサイズとVRAMの最適化**:
    Ollamaのコンテキストウィンドウを32,768（32K）に拡大し、プロンプト内のJSONデータから冗長なフィールドを削除することで、100曲近いアルバムでも情報の欠落（Truncation）を防ぎました。
*   **Direct JSON出力の強制**:
    最近のLLM（Qwen 3.5等）が過剰に「思考（Thinking）」を行い、出力バジェットを使い果たす問題を解決するため、システムプロンプトを「メタデータ監査エンジン」として純粋なJSON出力に特化させ、推論ブロックを抑制しました。
*   **安定モデルの選定**:
    推論速度と精度のバランスに優れた `qwen2.5:7b-sst` を標準モデルとして採用し、確実かつ高速なメタデータ統合を実現しました。

## 6. 現在の実装との比較・期待される効果

| 項目 | 現在の実装 (`src/scout/`) | 仮想アルバム構想導入後 |
| :--- | :--- | :--- |
| **同定 (Identity) の責任** | LLM（複数のMBZ候補から推論・選択） | Pythonロジック（AcoustIDクロスバリデーションで確定） |
| **LLMへの入力情報** | 大量のMBZ候補リストとSteam生データ | 完全に構造化された3つの仮想アルバム情報のみ |
| **LLMのタスク** | 同定とタグマージの同時処理 | フィールドごとのタグマージと最終調整のみ |
| **コンテキストサイズ** | 巨大（100曲超でトークン溢れの危険） | 軽量（確定情報のみを渡すため） |
| **処理精度・安定性** | 複数候補による迷いやハルシネーションが発生しやすい | 決定論的アプローチにより誤爆が理論上ほぼゼロになる |

### 結論

仮想アルバム構想の導入により、LLMの推論負荷とプロンプトサイズを劇的に削減でき、同時に「FINGERPRINTによる多数決」という決定論的ロジックによってシステムの同定精度と安定性が飛躍的に向上します。LLMは本来の強みである「表記揺れの吸収」や「優先順位に基づく最適なフィールド選択」に専念できるようになります。
