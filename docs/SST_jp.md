# S.S.T (Steam Soundtrack Tagger) - 決定版仕様書

本書は S.S.T プロジェクトにおける**唯一の真実（Single Source of Truth）**です。過去のドキュメントはすべて非推奨となりました。本書は、システムの挙動、メタデータの整合性、および処理パイプラインに関する鉄の掟を定義します。

---

## 1. 根本原理: 「一度のアーカイブで、一生の信頼を」
- **主要ターゲット**: 日本語話者。
- **目標**: 高精度、メンテナンスフリーのオーディオアーカイブ。
- **ポリシー**: メタデータの正確性や音源の整合性に少しでも疑念がある場合、そのアイテムは必ず `REVIEW` へ送られなければならない。`ARCHIVE` ステータスは、100% の信頼性を象徴するものでなければならない。

---

## 2. メタデータの主権（確定情報：LOCKED TRUTH）
特定の情報は「決定論的（Deterministic）」と見なされ、LLM やヒューリスティックによって変更することはできない。

### 2.1 Steam API データ (LOCKED)
Steam ストア API から得られる以下のフィールドは絶対的な真実である：
- **アルバムタイトル**: `steam_meta.name`
- **アーティスト (アルバムレベル)**: `steam_meta.developer` + `steam_meta.publisher`
- **リリース年**: `steam_meta.release_date[:4]`
- **Steam ID**: `app_id`, `parent_app_id`
- **コメント/グルーピング**: 親ゲーム（Parent Game）の詳細を参照しなければならない。`parent_app_id` が存在しない場合は、サウンドトラック自身の AppID/名称をフォールバックとして使用する。

### 2.2 ハイブリッド・スコアリング・システム
絶対的な信頼性を確保するため、システムは二段階の評価プロセスを採用する：

#### 第一段階：Python による数学的ふるい分け（`mbz.py` で計算）
Python は、物理的な証拠に基づいて各 MusicBrainz 候補の決定論的なスコアを計算する：
- **AppID 一致 (+500点)**: `url-rels` 内に Steam AppID への直接リンクがある。
- **親ゲーム AppID 一致 (+300点)**: `url-rels` 内に親ゲーム AppID へのリンクがある。
- **Bandcamp ボーナス (+100点)**: `url-rels` 内に公式 Bandcamp リンクがある。
- **タイトル類似度 (0〜+100点)**: Steam 名またはローカルタグに対する最高の `SequenceMatcher` スコア。
- **構造の一致 (+50点)**: トラック数の完全一致（1曲の乖離につき10点を減点）。
- **トラックリストの指紋 (+200点)**: 各曲名の平均類似度が 80% を超える場合に付与。

#### 第二段階：LLM による意味論的監査（`llm.py` で計算）
LLM は、事前にソートされたショートリスト（上位 3〜5 件）に対して最終的な裁定を下す：
- **翻訳マッチング**: 言語の違いを解決する（例：「Battle」と「戦闘」）。
- **タグの衛生状態**: "Dirty Tags"（番号混入など）やその他の微細な異常を特定する。
- **最終判決**: 最終的なランク（S、A、B、または C）を割り当てる。

### 2.3 ゲート方式スコアリングシステム
LLM の確信度スコアは、以下の厳格な閾値に従わなければならない：
- **Rank S (100%)**: Steam、MBZ、埋め込みタグの3者が曲名・曲数ともに完璧に一致。 -> **ARCHIVE**
- **Rank A (95%)**: 高い一貫性があり、"Dirty Tags"（番号混入など）が存在しない。 -> **ARCHIVE**
- **Rank B (80-90%)**: わずかな不一致、曲数の相違、または "Dirty Tags" が検出された場合。 -> **REVIEW**
- **Rank C (< 80%)**: 競合がある、または証拠が不十分。 -> **REVIEW**

### 2.3 Steam API アクセスの最適化
- **ターゲット・フェッチ**: システムは Steam API キーと `@data/userdata.json` を活用して、関連する AppID（ウィッシュリスト、所有ライブラリ等）を特定しなければならない。
- **最小限の負荷**: ストア API への総当たり的なスキャンは厳禁とする。ユーザーのアプリリストを使用して、確認済みのサウンドトラックまたは親ゲームのメタデータを、必要最小限の API 呼び出し回数でピンポイントに取得すること。

## 3. オーディオおよびパッケージングの整合性
### 3.1 ネイティブ・バッファリング戦略
Windows マウント (`/mnt/d`) からの I/O ジッターエラーを防ぐため：
1.  **収集 (Gather)**: 音源ファイルを WSL2 ネイティブファイルシステム (`/tmp/sst-work`) にコピーする。
2.  **変換 (Transform)**: ローカルファイルに対して FFmpeg を実行する（厳格な ID3v2.3、AIFF/MP3 320k）。
3.  **検証 (Validate)**: FFmpeg の警告またはエラーが検出された場合、**強制的に REVIEW** へ送る。

### 3.2 アトミック・ムーブ
ZIP ファイルは WSL2 ネイティブ領域で生成し、圧縮が 100% 完了し検証された後にのみ Windows 領域へ移動しなければならない。

## 4. LLM 運用契約（ゼロ・ハルシネーション）
- **思考言語**: 日本語 (JA) のみ。
- **推論の禁止**: 内部的なファイル名（例: `bgm_01.wav`）から曲名やアーティストを憶測（推測）することを禁ずる。
- **セマンティック・ラベリング**: Review アイテムに対して、LLM はデータ異常の具体的な内容（例：「BGM/SFX混在」「Dirty Tags検出」）を人間が読みやすいラベルで提供しなければならない。
- **ステートレス・チャンク**: 大規模なアルバムを分割処理する場合、フェーズ1で決定した `Global Identity` を絶対的な制約としてフェーズ2に注入しなければならない。

---

## 5. タギングフォーマット標準
- **セパレータ**: 複数値フィールドには `; `（セミコロン＋スペース）を使用する。
- **言語タグ (TLAN)**: ISO 639-2（例: `jpn`）を使用する。
- **フィールドマッピング**:
    - `TPE1` (アーティスト): MusicBrainz クレジット（最優先）または開発者（フォールバック）。
    - `TPE2` (アルバムアーティスト): 開発者; 出版社。
    - `TALB` (アルバム名): Steam アルバム名 (Locked)。
    - `COMM` (コメント): 親ゲーム名; AppID; ストア URL。

---

## 6. システム実行モード
- **`production`**: ターミナルは静音（プログレスバーのみ）。ログはファイルへ出力。高スループット（CPU最適化）。
- **`development`**: コンソールに詳細ログを出力。エラーディレクトリを監査用に保持。キャッシュをバイパス。
- **`LLM_BACKEND`**:
    - **`OLLAMA`**: ローカルの Ollama インスタンスに最適化。パラメータ制御（例：`num_ctx: 32768`）の確実性のためにネイティブ `/api/chat` を使用。
    - **`GEMINI`**: Google AI Studio (Gemini 1.5 Pro/Flash) に最適化。
    - **`OPENAI_COMPATIBLE`**: LM Studio、または OpenAI 本家向けの汎用モード。標準の `/v1/chat/completions` を使用。
