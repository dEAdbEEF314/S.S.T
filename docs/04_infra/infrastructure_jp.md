# インフラ構成

## アーキテクチャ：マルチコンテナ戦略

S.S.Tシステムは、**独立し、疎結合化されたDockerコンテナ**の集合として設計されています。各コンポーネントは隔離された状態で実行可能であり、ネットワークプロトコル（HTTP/S3/API）を介して通信しなければなりません。

## ストレージ構造

```
sst/
├ ingest/
├ archive/
├ review/
└ processed/
```

---

## コンポーネント（独立コンテナ）

- **Docker / Docker Compose**: ローカルおよび本番環境での標準的なオーケストレーションツール。
- **Prefect**: 疎結合化されたオーケストレーションエンジン（Core）。
- **SeaweedFS**: S3互換のオブジェクトストレージ（Storage）。
- **SST Workers**: スケーラブルでステートレスな処理ユニット。

---

## 設計原則

- **独立したスケーリング**: 各コンテナは負荷に応じて水平方向にスケール可能です。
- **隔離**: 1つのコンテナの失敗が、他のコンテナの健全性に影響を与えてはなりません。
- **ステートレスなワーカー**: ワーカーは永続的なローカル状態を持ちません。
- **べき等な処理**: 同じ入力に対してコンテナを再実行すると、同じ出力が生成されます。

---

## ロギング

すべてのコンテナで必須のフィールド：

- job_id
- track_id
- step
- result
- error

---

## 実行

- **完全なコンテナベース**
- Prefectによるオーケストレーション（または単純なセットアップではDocker Compose）
- コンテナ内での依存関係の一貫性を保つため、`uv` を使用して管理
