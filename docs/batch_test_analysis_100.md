## 2. Review 送りとなった原因の分類と分析 (37件)

Review送りとなった要因は、処理フェーズごとに大きく **3つのパターン** に分類できます。

### 【パターンA】 LLM Phase 2 タイムアウトによる失敗 (3件 / 8%)
LLMによる「Phase 1 (Identity確認)」では高評価を得たものの、「Phase 2 (トラックマッピング)」の推論でLLMサーバーからの応答がタイムアウト（`read timeout=300` × 3回リトライ）し、マッピング情報が生成できなかったケースです。
* **該当タイトル**: 『Neon Abyss』, 『The Messenger Disc III』, 『Cyberpunk 2077 Bonus Content』

### 【パターンB】 LLM Phase 1 での低信頼度による早期リジェクト (13件 / 35%)
LLMが算出した `identity_confidence` (同一性信頼度) が閾値の **90未満** だったため、マッピングを行わずに直接Reviewへ送られたケースです。（※この層はDiscord通知をスキップする仕様に沿っています）
* **該当タイトル**: 『Happy’s Humble Burger Farm』シリーズ (11件), 『Batbarian』, 『Phantom Breaker: Omnia』
* **分析**: 『Happy's Humble Burger Farm』等はSteam上では数曲のDLCアルバムですが、ローカルファイル群が別アルバムや127曲構成の巨大なコンピレーションになっており、波形照合率（FINGERPRINT）が 15%〜30% と著しく低くなりました。LLMがこれを「同一アルバムではない」と正しく見抜き、低い信頼度を返しています。非常に優秀なガードレールとして機能しています。

### 【パターンC】 Validator (最終物理チェック) による降格 (21件 / 57%)
LLMの推論はすべて完了したものの、最後の `validator.py` による整合性チェックで問題が検出され、ArchiveからReviewへ降格したケースです。Discordへ警告通知が飛んだのはこの層です。
* **該当タイトル**: 『FlatOut 4』, 『30XX』, 『式神の城II』, 『Desperados III』 など21件。
* **主な検出理由**:
  1. **FINGERPRINTデータの品質不良・矛盾**: 波形照合はできたが、MusicBrainz側のデータが欠落していたり曲順がSteamと全く異なる。
  2. **非楽曲コンテンツの混在**: BGMだけでなくSE（効果音）が大量に含まれており、LLMの品質評価スコアがArchive基準（90以上）に届かなかった。
  3. **トラックリストの不整合**: タイトルの重複や、タグにゴミが残っているケース。

## 3. Archive 送り内容の妥当性検証 (63件)

「相応しくないArchive行きが含まれていないか」「システムとLLMの間に矛盾がないか」を重点的に検証しました。

### 3.1. なぜArchive行きになったか？ (到達条件)
今回Archiveへ到達した63件は、以下の「二段構えの審査」をすべて無傷で通過したエリートデータのみです。
1. **LLMの知的審査**: `identity_confidence` が 95% 以上、かつ `integrity_quality` が 90% 以上であり、LLMがArchiveを推奨している。
2. **システムの物理審査**: `validator.py` において、Track番号「0」や「Unknown Title」、タイトルの重複、エンコードエラーなどの物理的な欠陥が一切ない。

### 3.2. Archive行きに相応しくない物は含まれていないか？
全63件のメタデータを直接解析した結果、**相応しくない（基準を満たしていない）Archive行きは1件も存在しませんでした。**

特筆すべきは、FINGERPRINT（波形照合）が `0.0%` や `2.6%` と絶望的でMusicBrainzが全く関係ない版を誤検知していたケース（AppID: `1702020` 等）です。これらにおいて、LLMは「FINGERPRINTは誤検知だが、Steam公式のトラックリストとローカルファイルが構成・曲名ともに完全一致している」と推論し、プロンプトの `STEAM-TRUST PATH` ルールを適用して自ら自信を持ってスコア100を出しArchiveへ導いていました。これは誤検知のすり抜けではなく、意図された設計通りの見事なリカバリー処理です。

### 3.3. システム処理とLLM回答に矛盾しているものはないか？
**矛盾は一切ありませんでした。**
* LLMが「Reviewにすべき」と言ったのにシステムが勝手にArchiveしたケースは **0件** です。
* 逆に、LLMが「Archiveできる」と満点を出しても、システム側（Validator）が「一部の曲名に重複やゴミがある」と物理的に検出した場合は、LLMの判断を無視して安全側に倒し（ダウングレード）、Reviewディレクトリに退避させたケースが **21件**（パターンC）存在します。
