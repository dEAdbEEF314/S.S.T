# SST パイプライン実行戦略

## 概要
SST パイプラインは、分散型の「Flow-of-Flows（フロー・オブ・フロー）」アーキテクチャを採用しています。これにより、オーケストレーション論理（Core）と重い処理論理（Worker）が分離され、独立したスケーリングとリモート実行が可能になります。

## パイプラインのフロー

1.  **フェーズ 1: スカウト (Scout ノード)**
    - ローカルの Steam ライブラリをスキャンしてサウンドトラックを探します。
    - 生ファイルと `.acf` マニフェストを S3 (`ingest/` バケット) にアップロードします。
    - REST API の POST リクエストを介して Prefect の **メインフロー** をトリガーします。

2.  **フェーズ 2: オーケストレーション (Core ノード)**
    - スカウト結果を受信します。
    - アルバムのリストをループ処理します。
    - 各アルバムに対して `run_deployment("sst-worker-flow/sst-worker-deployment")` を呼び出します。
    - これにより、Worker によってピックアップされるサブフローの実行が作成されます。

3.  **フェーズ 3: 処理 (Worker ノード)**
    - Worker フローサーバーが割り当てられた `sst-worker-flow` をピックアップします。
    - **ロジック**:
        - ファイルを S3 から一時ストレージにダウンロードします。
        - MusicBrainz と埋め込みメタデータを使用してアルバムを識別します。
        - **AI インタラクション**: タイトルの正規化とメタデータ検証のために LLM を呼び出します。
        - **タグ付け**: 必要に応じてオーディオ形式を正規化し、ID3/FLAC タグとアートワークを適用します。
        - **アップロード**: タグ付けされたファイルを `archive/` または `review/` にプッシュし、LLM とのやり取りを `logs/llm/` にログ記録します。

## 信頼性とモニタリング
- **エラー処理**: Worker フローの失敗によってメインフローがクラッシュすることはありません。ダッシュボードで個々のアルバムの状態を確認できます。
- **可観測性**: すべてのやり取りは S3 に記録され、S.S.T ダッシュボード UI を介して閲覧できます。
- **並列性**: 異なるアルバムを並行して処理するために、複数の Worker を展開できます。
