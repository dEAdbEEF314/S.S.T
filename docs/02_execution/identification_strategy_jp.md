# 同定およびタイブレーク戦略 (Act-11)

SST は、メタデータの矛盾を解決するために多層的な証拠アプローチを採用しています。このドキュメントでは、システムがどのように候補をランク付けし、情報を重み付けするかを説明します。

## 1. 情報の証拠の重み

メタデータを統合する際、システムは各ソースに以下の重みを割り当てます。

| 重みのレベル | ソースの種類 | 説明 |
| :--- | :--- | :--- |
| **LOCKED (固定)** | Steam API | AppID、ストア URL、開発元などの固定値。AI による変更不可。 |
| **STRONG (強)** | 相互検証済みタグ | 複数のファイル形式（例：MP3 と FLAC）で完全に一致したメタデータ。 |
| **BASE (基本)** | MusicBrainz | 公式データベースの候補。論理的なトラックリスト構造を提供。 |
| **WEAK (弱)** | 生ヘッダー / ファイル名 | 単一のファイルから抽出された、またはファイル名から推測されたデータ。 |

## 2. MusicBrainz スコアリング (ルール a-f)

システムは、LLM に提示する前に MusicBrainz (MBZ) の候補をフィルタリングし、ランク付けします。スコアは以下のように計算されます。

- **a. タイトル一致 (+40)**: MBZ のリリース・タイトルが Steam のサントラ名と一致。
- **b. フォーマット = "Digital Media" (+50)**: デジタル・リリースを最優先。
- **c. 文脈: "Bandcamp" なし (+20)**: Bandcamp 特有のタグ付けスタイルを回避。
- **d. トラック数の一致 (+40 または +20)**:
    - ローカルの物理ファイル数と完全一致: +40
    - ±2 トラックの誤差: +20
- **e. サントラ発売日 (+40 または +20)**:
    - 年が Steam のサントラ発売日と一致: +40
    - 誤差 ±1 年以内: +20
- **f. 親ゲーム発売日 (+30 または +10)**:
    - 年が親ゲームの発売日と一致: +30
    - 誤差 ±1 年以内: +10

**ステータス・ペナルティ**: "Bootleg"（海賊版）リリースには -100 のペナルティが課されます。

## 3. 統合ワークフロー

1.  **ランキング**: Python の前処理プログラムが上記のルールに基づいて MBZ 候補をランク付けします。
2.  **スリミング (剪定)**: スコア上位 3〜5 件の候補（MBID とタイトル）のみを LLM に送信します。
3.  **クロスリファレンス**: LLM は選択された MBZ 候補を「強い証拠」（ローカル・タグ）と比較します。
4.  **自信度の計算**:
    - **100点**: 強い証拠とトップ MBZ 候補が完全に一致。
    - **80-90点**: 些細なタイトルの違い（例：接尾辞 "Remix" の有無）や日付の不一致を解決。
    - **80点未満**: トラック数、アーティストの深刻な不一致、またはデータの欠落。（`review/` 振り分けの対象）。
