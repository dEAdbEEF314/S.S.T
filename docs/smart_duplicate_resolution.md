# Smart Duplicate Resolution Specification

## 1. 概要 (Overview)
S.S.T (Steam Soundtrack Tagger) では、サウンドトラック特有の「同一曲名の別バージョン（インスト版、ショート版等）」や「Steam側のメタデータ不備による同名トラックの連続」といった問題に対し、単なるLLMへの丸投げではなく、プログラムによる決定論的な解決アルゴリズム（Smart Duplicate Resolution）を実装しています。

このロジックは、LLMの推論結果に複数のローカルトラックが「同一のSteamトラック（同じ `matched_v_idx`）」に割り当てられた場合（衝突・重複）に発動し、安全かつ論理的に正しいトラックへ再分配を行います。

## 2. 実装モジュール
コアロジックは `src/sst/processor.py` の `_resolve_duplicate_mappings` メソッドに実装されています。

## 3. 重複解決の3つのヒューリスティック (Heuristics)

LLMが複数のローカルトラック群に対して同じ `v_idx` を返してきた場合、以下の3つのヒューリスティック（経験則）を順番に適用し、衝突の解消を試みます。

### Heuristic 1: Multi-disc Mismatch (ディスク番号の不一致による衝突)
LLMがディスク構成を誤認し、ローカルの「Disc 1のTrack 1」と「Disc 2のTrack 1」を、Steam側の「Disc 1のTrack 1」にまとめてマッピングしてしまうケースを救済します。
* **動作**: 衝突したローカルトラックのディスク番号 (`l_disc`) が複数混在しているかチェックします。混在している場合、それぞれのローカルディスク番号とSteam側のディスク番号が一致し、かつタイトルが類似している別の `v_idx` を探し出し、そこに再マッピングします。

### Heuristic 2: Name-based Recovery (同一ディスク内でのファジーネーム検索)
同一ディスク内において、LLMが「タイトルは違うが、関連性がある」と誤認して複数の異なる曲を同じ `v_idx` に割り当ててしまったケースを救済します。（例: Happy's Humble Burger Farm などで顕著）
* **動作**: 衝突した各ローカルトラックのタイトル名を用いて、Steamトラックリスト（同一ディスク内）をファジー検索（前方一致・完全一致）します。よりタイトルが合致する別の `v_idx` が見つかった場合、そちらへ独立して引き剥がします。

### Heuristic 3: Consecutive Same-name Sequence (連番同名トラックへの再分配)
Steam側のメタデータ自体に「BGM」「BGM」「BGM」のように全く同じ曲名が連続して登録されている場合、LLMは全てのローカルトラックを「最初のBGM (v_idx)」にマッピングしがちです。
* **動作**: 衝突が発生した `v_idx` を起点に、Steam側のトラックリストで「同じ曲名がどこまで連続しているか」をカウントします。連続する枠（Sequence）が、衝突しているローカルトラックの数以上あれば、ローカル側の順番（ファイル名ソート順）に従って、連続するSteamインデックスへ順番に1つずつ割り当て直します。

## 4. 最終フェイルセーフ (Validator)
上記3つのヒューリスティックを経てもなお、複数のローカルトラックが同じ「Disc番号とTrack番号」を共有している場合（物理的な重複）、システムは最終の物理審査を行います。

`src/sst/validator.py` の `validate()` 内にて、生成された最終タグ情報を全トラック分スキャンし、`duplicate_pairs`（重複トラック）を検出します。
もし重複が残存していた場合は、**安全装置（フェイルセーフ）が発動し、`issues` に "Duplicates" を追加して Archive処理を即座に中止し、強制的に Review ディレクトリへ降格させます。**

これにより、同一アルバム内に同じトラック番号を持つMP3が複数生成されてしまうデータ破壊事故を100%防いでいます。
