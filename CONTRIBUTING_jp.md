
# 貢献ガイド (人間 & AI)

このプロジェクトは **AIによる補助的な実装** を前提として設計されています。

すべての貢献者（人間またはAIエージェント）は、このドキュメントに従わなければなりません。

---

# 🧭 基本原則

1. **速度より正確性**
2. **サイレントな失敗の禁止**
3. **決定論的な挙動**
4. **仕様優先の実装**
5. **低信頼度 → レビューへ（決して推測しない）**

---

# 📚 信頼できる情報源（優先順位）

以下の優先順位を遵守しなければなりません：

1. `docs/01_spec/*`（厳密なコントラクト）
2. `docs/02_execution/*`（振る舞いのルール）
3. `docs/03_ai/*`（AIの判断ルール）
4. `docs/00_overview/*`（コンテキストのみ）

矛盾がある場合：
→ **仕様（Spec）がすべてに優先します**

---

# 🚫 厳禁事項

## ❌ ハルシネーション（捏造）の禁止
- メタデータを捏造しない
- 不足しているフィールドを勝手に補完しない
- 推測で「ギャップを埋める」ことをしない

## ❌ サイレントフォールバックの禁止
- すべてのフォールバックはログに記録されなければならない
- すべての機能低下は明示的でなければならない

## ❌ 仕様違反の禁止
- I/Oフォーマットを変更しない
- 必須フィールドを変更しない
- 必要なステップをスキップしない

## ❌ 隠れた状態（ステート）の禁止
- ワーカーはステートレスでなければならない
- 定義されていない限り、暗黙的なキャッシュを行わない

---

# ✅ 必須とされる振る舞い

## 決定論的な処理
- 同じ入力 → 同じ出力
- 明示的に許可されていない限り、ランダム性を持たせない

## 明示的な失敗ハンドリング
- すべての失敗は以下を生成しなければならない：
  - エラーログ
  - status = "review" または "failed"

## ロギング（必須）
各ステップは必ず以下を記録しなければならない：

- job_id
- track_id
- step
- result
- error (発生した場合)

---

# 🔁 リトライルール

- 一時的なエラーのみリトライする
- 指数関数的バックオフを使用する
- 設定（config）のリトライ制限を遵守する

禁止事項：
- 無限リトライ
- 決定論的な失敗（バグ等）に対するリトライ

---

# 🧠 AI固有のルール

AIエージェントは以下を遵守しなければならない：

- `docs/03_ai/agent_prompt.md` に従う
- 有効な場合はデュアルプロバイダーによる検証を行う
- 信頼度しきい値を遵守する

### 判断ルール

| 条件 | アクション |
|----------|--------|
| 高信頼度 | 承認 |
| 中信頼度 | 警告付きで承認 |
| 低信頼度 | レビューへ送る |

---

# 📦 出力コントラクト

すべての出力は以下に従わなければならない：

👉 `docs/01_spec/io_spec.md`

要件：

- 有効なJSON
- すべての必須フィールドが存在する
- パスが有効である
- 不完全な構造を出力しない

---

# 🧪 テスト要件

マージ前に以下を確認：

- JSONスキーマの検証
- 少なくとも以下のケースをテスト：
  - 成功ケース
  - フォールバックケース
  - 失敗ケース

---

# 🧩 コードスタイル & 環境

- 言語: Python 3.11+
- **環境管理: `uv` の使用が必須です。**
  - すべての仮想環境は `uv` を介して作成・管理されなければなりません。
  - 依存関係のロックは `uv` で行わなければなりません。
- ステートレスな関数を推奨
- 可能な限り純粋関数を使用
- サイドエフェクト（副作用）は明示的であること

---

# 🔀 ブランチ / コミットルール

## コミットメッセージ形式

```
[type] short description

例:
feat: add vgmdb client
fix: handle acoustid timeout
refactor: split pipeline step
```

## ルール

- 1つの論理的な変更につき1つのコミット
- 複数の関心事を混ぜない
- 再現可能であること

---

# 🧱 プルリクエストルール

PRには以下を含めること：

- 何を実装したか
- どの仕様（spec）に従ったか
- テスト結果

---

# ⚠️ 失敗のルーティング

以下のいずれかが発生した場合：

- 信頼度 < しきい値
- データの不一致
- APIの不整合
- タグ付けの不確実性

→ 必ず以下のようにルーティングすること：

```
status = "review"
```
